# ADR-006: systems.yaml — ECS-Style System Declaration

**Status:** Accepted
**Date:** June 2026
**Supersedes:** None
**Related:** ADR-001 (three-file format), ADR-002 (Lua logic layer), ADR-005 (systems as patterns)

---

## Context

ADR-005 established that component systems are named patterns, not shared binaries.
Each game implements the systems it needs in its own logic.lua. This was correct for
the first game.

As the studio adds games, a new problem emerges: different games use different system
subsets, and there is no machine-readable declaration of which systems a game uses.
The runtime loads all of logic.lua regardless of what the game actually needs.
Claude cannot know what systems a game uses without reading the full Lua file.

The horse racing game uses: GeneticsSystem, OddsSystem, BreedingSystem, MarketSystem.
The BattleBots game will use: DurabilitySystem, CombatSystem, MarketSystem, ArenaSystem.
A cooking competition game would use: QualitySystem, IngredientSystem, JudgingSystem.

The studio needs a way to declare what a game is made of — before the runtime loads it,
before Claude reads it, and before a second developer tries to understand it.

---

## Decision

Every game in the studio declares a fifth file: `systems.yaml`.

```
games/
  [game_id]/
    data.yaml
    ui.yaml
    logic.lua
    systems.yaml    ← NEW
```

`systems.yaml` is an ECS-style manifest. It declares:
- Which systems the game uses
- Which components each system operates on
- Which entities exist in the game
- Which components each entity has
- Which Lua functions implement each system

The runtime reads `systems.yaml` before loading `logic.lua`. Claude reads it first
when reasoning about an unfamiliar game.

---

## Format

```yaml
# systems.yaml — [game_id]
engine_version: "1.0"

systems:
  - id: genetics
    description: "Genome generation, trait inheritance, breeding"
    components: [genome, generation, traits]
    functions:
      - generate_horse
      - breed_horses
      - breed_stat

  - id: odds
    description: "Race odds calculation from stat profiles"
    components: [speed, stamina, acceleration, temperament, distance]
    functions:
      - calculate_odds
      - calculate_place_odds
      - calculate_show_odds

  - id: simulation
    description: "Race physics — headless and per-tick"
    components: [position, velocity, energy, finish_time]
    functions:
      - simulate_race
      - tick_race

  - id: market
    description: "Bet placement, settlement, horse valuation"
    components: [bet, payout_odds, prize_pool, stake]
    functions:
      - settle_bets
      - calculate_horse_price
      - sell_horse
      - update_horse_after_race

  - id: breeding
    description: "Pairing validation, offspring generation, cost calculation"
    components: [genome, generation, parents, cooldown]
    functions:
      - breed_horses

entities:
  horse:
    description: "A racing horse with stats, career, and lineage"
    systems: [genetics, odds, simulation, market, breeding]
    schema_ref: data.yaml#horse

  race:
    description: "A race event with participants, distance, and prize"
    systems: [simulation, odds, market]
    schema_ref: data.yaml#race_classes

  bet:
    description: "A wager on a race participant"
    systems: [market]
    schema_ref: data.yaml#bet
```

## Rules

**Rule 1: systems.yaml is declarative, not imperative.**
It describes what the game uses. It does not control load order or initialization.
The runtime still loads all of logic.lua. systems.yaml is a manifest, not a module loader.

**Rule 2: Every function in logic.lua must appear in exactly one system's `functions` list.**
If a function is not listed, it is either dead code or the systems.yaml is incomplete.
Both are errors. The studio linter (Phase 4) will enforce this.

**Rule 3: Every entity must declare its systems.**
An entity with no systems declared cannot be used by the runtime bridge. This prevents
silent failures where an entity type is referenced but has no defined behavior.

**Rule 4: Claude reads systems.yaml first.**
When Claude is given a new game to reason about, systems.yaml is the entry point.
It answers "what does this game do" before logic.lua answers "how does it do it."

**Rule 5: systems.yaml is locked by ADR.**
Adding a new system to an existing game requires an ADR. Renaming a system requires
an ADR. Removing a system requires an ADR. Adding new functions to an existing
system does not require an ADR — only updating systems.yaml.

**Rule 6: Component names are snake_case nouns.**
`speed` not `speedStat`. `finish_time` not `finishTime`. `prize_pool` not `prizePool`.
Same convention as all other field names in the studio.

## Four-File Contract (updated from ADR-001)

Every game in the studio is now defined by exactly four files:

```
games/[game_id]/
  data.yaml       — entities, constants, tables, schemas
  ui.yaml         — layout intent (renderer-agnostic)
  logic.lua       — pure logic, no I/O, no rendering
  systems.yaml    — ECS manifest: systems, components, entities
```

ADR-001's guarantee extends: any game defined in four files runs on any supported
runtime. Claude can read four files and accurately describe the game's systems,
entities, and behavior.

## The BattleBots systems.yaml (preview)

```yaml
engine_version: "1.0"

systems:
  - id: durability
    description: "Part health, damage thresholds, destruction"
    components: [part, health, damage_threshold, armor]
    functions:
      - apply_damage
      - check_part_destroyed
      - calculate_bot_effectiveness

  - id: combat
    description: "Attack resolution, hit probability, damage calculation"
    components: [attack, defense, reach, speed, weapon_type]
    functions:
      - resolve_hit
      - calculate_damage
      - simulate_fight

  - id: live_market
    description: "Real-time odds shifting on damage events"
    components: [odds, damage_state, bet, lock_time]
    functions:
      - recalculate_odds_on_damage
      - settle_bets
      - lock_bet

  - id: arena
    description: "Four simultaneous bouts, fight scheduling"
    components: [arena_id, bout, participants, round]
    functions:
      - create_card
      - assign_bouts
      - resolve_card

entities:
  bot:
    systems: [durability, combat]
  part:
    systems: [durability]
  bet:
    systems: [live_market]
  bout:
    systems: [arena, combat]
  card:
    systems: [arena, live_market]
```

The horse racing and BattleBots games share `market`/`live_market` as a system
family — the bet mechanics are structurally identical, just operating on different
timescales and resolution triggers.

## Consequences

**Positive:**
- Claude can understand any game from four files without reading logic.lua first
- The runtime bridge can validate that all declared functions exist before loading
- The studio linter (Phase 4) has a ground truth to lint against
- New developers understand game scope from systems.yaml before touching code
- BattleBots and future games have a defined place to declare their systems

**Negative:**
- systems.yaml must be kept in sync with logic.lua manually until Phase 4 linter ships
- Adding a new system requires updating systems.yaml, logic.lua, and an ADR — three
  touchpoints instead of one

**Neutral:**
- The runtime does not change. systems.yaml is read-only metadata in Phase 3.
  Enforcement is a Phase 4 deliverable.

---

## Implementation

**Immediate:** Write `games/horse_racing/systems.yaml` using the format above.
This locks the horse racing game's system declarations permanently.

**Phase 4:** Studio linter reads systems.yaml and verifies every function in
logic.lua appears in exactly one system's `functions` list. Fails the build if
systems.yaml is stale.

**Phase 4:** Runtime bridge reads systems.yaml and exposes `get_systems()` and
`get_entities()` as MCP-callable tools.
