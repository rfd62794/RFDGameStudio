# ADR-008: UI Component Vocabulary

**Status:** Accepted
**Date:** June 2026
**Supersedes:** None
**Related:** ADR-001 (three-file format), SDD v0.1 §2.2 (ui.yaml), SDD v0.2 §10

---

## Context

The current TypeScript project has all UI code in three locations:

1. `ts/src/components/` — five horse-racing-specific components
2. `ts/src/index.css` — ~400 lines mixing design tokens, base resets, and
   horse-racing-specific styles in one flat file
3. `ts/src/App.tsx` — horse-racing-specific app shell

When Snake is added, it needs to render a grid, a score panel, and a game-over
screen. The current structure offers no reusable component or token layer. Snake
will either duplicate horse_racing styles or import them incorrectly.

VoidDrift's ADR-019 established the correct hierarchy for Bevy/egui:
Level 1 (Primitives), Level 2 (Components), Level 3 (Screens). This ADR
establishes the equivalent for RFDGameStudio's TypeScript layer.

The ui.yaml contract already separates layout intent from renderer implementation.
This ADR closes the remaining gap: the renderer itself needs a component hierarchy.

---

## Decision

### Three CSS Layers

```
ts/src/ui/tokens.css        — Layer 1: custom properties only, no selectors
ts/src/ui/base.css          — Layer 2: reset + typography using token variables
ts/src/games/{id}/styles.css — Layer 3: game-specific overrides and additions
```

**Layer 1 (tokens.css) rules:**
- Contains ONLY CSS custom property declarations inside `:root {}`
- No selectors other than `:root`
- No layout, typography, or component styles
- All games import this file. None may modify it.
- Token namespaces: `--color-*`, `--space-*`, `--font-*`, `--radius-*`

**Layer 2 (base.css) rules:**
- Reset (`*, *::before, *::after { box-sizing: border-box; }`)
- Body font, background, color from tokens
- Heading scales using `--font-size-*`
- No component class names
- No game-specific class names
- No `--horse-*`, `--snake-*`, `--bot-*` references

**Layer 3 (styles.css per game) rules:**
- May define game-specific class names (`.horse-card`, `.snake-cell`)
- May define game-specific token overrides (`.horse-card { --card-accent: var(--color-yellow); }`)
- May NOT redefine `--color-*`, `--space-*`, `--font-*`, `--radius-*` at `:root` level
- Imported AFTER `tokens.css` and `base.css`

### Three Component Layers

#### Layer 1 — Base Components (`ts/src/ui/components/`)

Stateless TypeScript functions. No game knowledge. No entity types.
Read tokens only. No inline styles.

**Mandatory registry (build order for Phase 2f):**

```typescript
// Button.tsx
interface ButtonProps {
  label: string;
  onClick: () => void;
  variant?: 'primary' | 'secondary' | 'danger' | 'neutral';
  disabled?: boolean;
  size?: 'sm' | 'md' | 'lg';
}
export function Button({ label, onClick, variant = 'primary', disabled, size = 'md' }: ButtonProps)

// Panel.tsx
interface PanelProps {
  children: React.ReactNode;
  className?: string;
  padding?: 'sm' | 'md' | 'lg';
}
export function Panel({ children, className, padding = 'md' }: PanelProps)

// StatBar.tsx
interface StatBarProps {
  label: string;
  value: number;
  max?: number;
  color?: string;    // CSS color value or token reference
  showValue?: boolean;
}
export function StatBar({ label, value, max = 100, color, showValue = true }: StatBarProps)

// Badge.tsx
interface BadgeProps {
  label: string;
  variant?: 'accent' | 'green' | 'red' | 'yellow' | 'amber' | 'muted';
}
export function Badge({ label, variant = 'muted' }: BadgeProps)

// TabBar.tsx
interface Tab { id: string; label: string; }
interface TabBarProps {
  tabs: Tab[];
  active: string;
  onSelect: (id: string) => void;
  variant?: 'default' | 'mobile';
}
export function TabBar({ tabs, active, onSelect, variant = 'default' }: TabBarProps)

// Card.tsx
interface CardProps {
  children: React.ReactNode;
  className?: string;
  onClick?: () => void;
}
export function Card({ children, className, onClick }: CardProps)

// Modal.tsx
interface ModalProps {
  title: string;
  children: React.ReactNode;
  onClose: () => void;
}
export function Modal({ title, children, onClose }: ModalProps)

// EmptyState.tsx
interface EmptyStateProps { message: string; }
export function EmptyState({ message }: EmptyStateProps)

// ErrorBox.tsx
interface ErrorBoxProps { message: string; }
export function ErrorBox({ message }: ErrorBoxProps)
```

**Rules for base components:**
- No imports from `../games/*`
- No imports from `../../engine/types` for game-specific types
- May import `React`, `framer-motion`, `lucide-react`
- Styles via CSS class names defined in `base.css` or `tokens.css`
- Never inline styles with hardcoded color values
- Always export as named export, never default

#### Layer 2 — Game Components (`ts/src/games/{id}/components/`)

Built from Layer 1 base components. Know about game entity types.
Import from `ts/src/ui/components/` and `ts/src/engine/`.

**Rules for game components:**
- May import base components: `import { Button, Panel, StatBar } from '../../ui/components'`
- May import engine types: `import type { Horse, GameSession } from '../../engine/types'`
- May NOT import from another game: no `import ... from '../horse_racing/components/...'`
- No game logic — calls Lua via runtime, renders result
- Styles in `ts/src/games/{id}/styles.css`, not inline

#### Layer 3 — Game Shell (`ts/src/games/{id}/App.tsx`)

The app shell for a specific game. Owns game state. Wires all handlers.
Renders game components. Manages persistence.

**Rules for game shell:**
- One per game
- All game state lives here (funds, entities, current match, history)
- All Lua calls for non-component logic originate here
- Passes everything down as props (no context, no global state)
- Imports from `ts/src/ui/components/` for structural elements (header, footer, tabs)

### Router (`ts/src/main.tsx`)

When more than one game exists, `main.tsx` becomes the game selector.

```typescript
// main.tsx — game selector
const GAMES = [
  { id: 'horse_racing', label: 'Derby Sim', component: HorseRacingApp },
  { id: 'snake',        label: 'Snake',     component: SnakeApp },
];

// Reads ?game= URL param or shows game selector screen
```

**Rule:** `main.tsx` imports game shells by name. Game shells do not import each
other. The router is the only file that knows about all games.

### ui.yaml Component Type Vocabulary

The shared `ui.yaml` component types (`type:` field) form a closed vocabulary.
Renderers implement these types. Game-specific types may be added only via ADR.

**Shared vocabulary (all games):**

| type | Renders |
|---|---|
| `label` | Text display |
| `stat_bar` | Labeled progress bar |
| `stat_display` | Numeric stat with label |
| `action_button` | Clickable button wired to an event |
| `card_grid` | Responsive grid of card items |
| `card` | Single entity card |
| `badge` | Small colored tag |
| `panel` | Surface container |
| `tab_bar` | Tab navigation |
| `empty_state` | Placeholder for empty collections |
| `data_table` | Tabular data display |
| `history_list` | Ordered list of past events |

**Game-specific types** (horse_racing only — not for import by other games):

| type | Renders |
|---|---|
| `horse_card` | Horse with SVGRacer sprite + stats |
| `race_track` | 6-lane animated SVG track |
| `betting_panel` | Win/Place/Show bet inputs |
| `odds_display` | Decimal odds badge |
| `breeder_selector` | Sire/dam pairing UI |

**Rule:** Game-specific types are prefixed with the game ID in systems.yaml but
are named plainly in ui.yaml. The renderer handles mapping. A new game that tries
to use `horse_card` type will receive an error — that type is not in the shared
vocabulary.

## Migration Path

This ADR takes effect immediately. The migration happens in Phase 2f.

**Phase 2f migration order:**

1. Create `ts/src/ui/` directory structure
2. Split `index.css` → `tokens.css` + `base.css` + `horse_racing/styles.css`
3. Create base components one at a time, replacing inline equivalents
4. Move horse_racing components to `ts/src/games/horse_racing/components/`
5. Move `App.tsx` to `ts/src/games/horse_racing/App.tsx`
6. Create `ts/src/main.tsx` as game router stub (horse_racing only for now)
7. Verify floors hold at each step before the next

**Phase 2f does NOT include:**
- Snake components (those are built with Snake)
- Complete replacement of all inline styles (incrementally per component)
- Full base component library (only what horse_racing already uses)

---

## Consequences

**Positive:**
- Snake renders from Day 1 using `<StatBar>`, `<Panel>`, `<Button>` — no duplicate CSS
- Design token changes propagate to all games from one file
- Any new game starts with a complete visual foundation, not from scratch
- Claude can generate a new game's components using only the base component registry

**Negative:**
- Phase 2f is a non-trivial refactor — existing tests must pass throughout
- The base component library must be built before it can be used (chicken-and-egg
  mitigated by doing it during horse_racing migration, not before)

**Constraint:**
- `index.css` must not be deleted until all consumers are migrated to the new
  location. Keep it as a re-export shim during the transition period.
