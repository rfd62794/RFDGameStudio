# ADR-005: Component Systems Are Named Patterns, Not Shared Binaries

**Status:** Accepted
**Date:** June 2026

## Context

The studio extracted eight recurring systems from five years of prior projects:
`GeneticsSystem`, `StatSystem`, `OddsSystem`, `DispatchSystem`, `ResourceSystem`,
`EventBus`, `CultureSystem`, `BreedingSystem`.

The temptation was to build a shared Lua library and `require()` it from each
game's `logic.lua`. This would create:
- Version coupling between games and the library
- A dependency graph that breaks the three-file contract (the game is no longer
  self-contained)
- A shared binary that makes porting to new runtimes harder

## Decision

Component systems are **named patterns** documented in the studio SDD.
Each game implements the patterns it needs in its own `logic.lua`.
No shared binary. No `require()`. The names exist to communicate design intent.

## Consequences

- Two games implementing `GeneticsSystem` will have different Lua code. That is
  correct — the pattern is the guarantee, not the implementation.
- The SDD registry table (§3) is the authoritative list of named patterns.
- When a new game implements a known pattern, the implementation should match the
  pattern's documented contract (inputs, outputs, purity) even if the code differs.
- This decision is permanent. Any proposal to create a shared Lua library requires
  a new ADR that explicitly supersedes this one.
