# ADR-001: Three-File Format Is the Canonical Game Definition

**Status:** Accepted
**Date:** June 2026

## Context

Prior projects defined games entirely in code:
- `rpgCore`: Python dataclasses and service classes
- `OperatorGame`: Rust structs and trait impls
- `horse_racing`: TypeScript React components

Porting between languages required full rewrites. The logic, the data, and the
rendering were entangled. There was no portable artifact representing "the game."

## Decision

Every game in RFDGameStudio is defined by exactly three files:

```
games/{game_id}/
  data.yaml     — entities, constants, tables, schemas
  ui.yaml       — layout intent (renderer-agnostic)
  logic.lua     — pure logic, no I/O, no rendering
```

These three files **are** the game. Runtimes implement them. No exceptions.

## Consequences

- New games require all three files before any runtime or renderer exists.
- Runtimes are forbidden from implementing game rules outside `logic.lua`.
- The format is locked — new top-level sections require a new ADR, not a silent
  schema change.
- Claude can read the three files and reason about the game without running code.
- Porting to a new runtime requires implementing the bridge only, never touching
  the game files.
