# ADR-007: Primitive Registry — Naming Conventions and Lua Contracts

**Status:** Accepted
**Date:** June 2026
**Supersedes:** None
**Related:** ADR-002 (Lua logic layer), ADR-005 (systems as patterns), ADR-006 (systems.yaml), SDD v0.2 §8–9

---

## Context

The seven game primitives were identified by translating Game Maker Studio's object
model (Create/Step/Draw/Collision/Destroy) into language-agnostic terms. The naming
convention across horse_racing/logic.lua is inconsistent — some functions use verb-noun
(`calculate_odds`), others use noun-verb (`horse_update`), others are unclear
(`tick_race` — is this Step or Movement?).

As the studio adds games, functions that implement the same primitive will be written
independently and incompatibly unless the contract is locked now.

This ADR locks:
1. Naming conventions for each primitive category
2. The Lua function signature contract per category
3. The `engine/` directory structure
4. Which existing horse_racing functions map to which primitive

---

## Decision

### Naming Convention by Primitive

Each function in the studio follows a verb-first naming convention. The verb
signals which primitive the function implements.

| Primitive | Verb(s) | Example |
|---|---|---|
| Entity | `generate_`, `create_`, `destroy_` | `generate_horse`, `create_snake` |
| Action | `apply_`, `update_`, `breed_` | `update_horse_after_race`, `apply_damage` |
| Resolution | `resolve_`, `simulate_` | `simulate_race`, `resolve_fight` |
| Consequence | `apply_`, `settle_` | `settle_bets`, `apply_race_result` |
| Movement | `tick_`, `move_` | `tick_race`, `move_snake` |
| Physics | `detect_`, `collide_` | `detect_wall_collision`, `collide_parts` |
| Lifecycle | `init_`, `step_`, `destroy_` | `init_game`, `step_game`, `destroy_entity` |

**Rule:** The verb is the contract. A function starting with `resolve_` must take
participants and return ranked outcomes. A function starting with `tick_` must take
an entity and a delta_time and return an updated entity.

### Engine Directory Structure

```
engine/
  primitives/
    entity.lua       — ID generation, schema validation helpers
    action.lua       — Pure state transformation utilities
    resolution.lua   — RNG-seeded outcome patterns
    consequence.lua  — Post-resolution state application utilities
    movement.lua     — Position update patterns (grid, continuous)
    physics.lua      — Collision detection/response patterns
    lifecycle.lua    — Lifecycle hook registry and dispatch
  systems/
    genetics.lua     — breed_stat, generate_color_profile, inheritance patterns
    odds.lua         — calculate_odds, calculate_place_odds, calculate_show_odds
    market.lua       — settle_bets, calculate_horse_price, sell_entity
    combat.lua       — resolve_hit, calculate_damage, apply_damage (BattleBots)
    grid_movement.lua — move_in_direction, detect_boundary, detect_self_collision
```

**Primitive files define contracts, not implementations.** They export pattern
functions and documented function signatures. System files implement those patterns
for specific domains.

### Primitive File Contracts

#### engine/primitives/entity.lua

```lua
-- Generate a unique ID for a new entity.
-- prefix: string identifying the entity type (e.g. "horse", "bot", "segment")
-- Returns: string ID
function generate_id(prefix)

-- Validate that an entity table has all required fields.
-- entity: table
-- required_fields: array of field name strings
-- Returns: true, nil | false, "Missing field: {name}"
function validate_entity(entity, required_fields)

-- Deep-copy an entity table (prevents mutation of inputs in actions).
-- entity: table
-- Returns: new table with same values
function copy_entity(entity)
```

#### engine/primitives/action.lua

```lua
-- Apply a delta to a numeric field on an entity.
-- Returns new entity with the field updated. Does not mutate input.
-- entity: table, field: string, delta: number
function increment_field(entity, field, delta)

-- Clamp a numeric field to [min_val, max_val].
-- Returns new entity. Does not mutate input.
function clamp_field(entity, field, min_val, max_val)
```

#### engine/primitives/resolution.lua

```lua
-- Score a set of participants by a stat-weight formula.
-- participants: array of tables with numeric stat fields
-- weights: {stat_name: weight} table
-- Returns: array of scores parallel to participants
function score_participants(participants, weights)

-- Convert raw scores to decimal odds with overround applied.
-- scores: array of numbers
-- overround: float (e.g. 1.12)
-- Returns: array of decimal odds
function scores_to_odds(scores, overround)
```

#### engine/primitives/movement.lua

```lua
-- Advance a continuous position by speed * delta_time.
-- Returns: new_position (float)
function advance_position(position, speed, delta_time)

-- Move a grid entity one step in a direction.
-- direction: {dx: int, dy: int}
-- Returns: {x: int, y: int}
function move_grid(pos, direction)

-- Check if a position is within grid bounds [0, width) x [0, height).
-- Returns: true if in bounds
function in_bounds(pos, width, height)
```

#### engine/primitives/physics.lua

```lua
-- Check if two grid positions are the same cell.
-- Returns: true if collision
function grid_collision(pos_a, pos_b)

-- Check if a position matches any position in an array (self-collision).
-- body: array of {x, y} tables
-- head: {x, y}
-- Returns: true if head hits body
function self_collision(head, body)
```

#### engine/primitives/lifecycle.lua

```lua
-- Standard lifecycle event names. Use as string keys in systems.yaml.
LIFECYCLE_CREATE    = "create"
LIFECYCLE_STEP      = "step"
LIFECYCLE_DRAW      = "draw"     -- TypeScript only, documented here for clarity
LIFECYCLE_COLLISION = "collision"
LIFECYCLE_DESTROY   = "destroy"

-- Validate that a game's systems.yaml declares functions for all lifecycle phases.
-- systems: parsed systems.yaml table
-- Returns: true, nil | false, "Missing lifecycle phase: {phase}"
function validate_lifecycle(systems)
```

### Existing Function Mapping

Every function in `games/horse_racing/logic.lua` maps to exactly one primitive.
This table is the canonical mapping. The ADR-007 linter (Phase 4) enforces it.

| Current Function | Primitive | Target Location |
|---|---|---|
| `clamp` | Action | `engine/primitives/action.lua` |
| `rand_int` | Resolution | `engine/primitives/resolution.lua` |
| `rand_item` | Resolution | `engine/primitives/resolution.lua` |
| `generate_id` (new) | Entity | `engine/primitives/entity.lua` |
| `copy_entity` (new) | Entity | `engine/primitives/entity.lua` |
| `generate_horse_name` | Entity | `engine/systems/genetics.lua` |
| `generate_color_profile` | Entity | `engine/systems/genetics.lua` |
| `generate_horse` | Entity | `engine/systems/genetics.lua` |
| `breed_stat` | Action | `engine/systems/genetics.lua` |
| `breed_horses` | Action | `engine/systems/genetics.lua` |
| `calculate_odds` | Resolution | `engine/systems/odds.lua` |
| `calculate_place_odds` | Resolution | `engine/systems/odds.lua` |
| `calculate_show_odds` | Resolution | `engine/systems/odds.lua` |
| `tick_race` | Movement | `games/horse_racing/logic.lua` (game-specific) |
| `simulate_race` | Resolution | `games/horse_racing/logic.lua` (game-specific) |
| `calculate_payouts` | Consequence | `engine/systems/market.lua` (deprecated, superseded by settle_bets) |
| `settle_bets` | Consequence | `engine/systems/market.lua` |
| `calculate_horse_price` | Consequence | `engine/systems/market.lua` |
| `sell_horse` | Action | `engine/systems/market.lua` |
| `update_horse_after_race` | Consequence | `games/horse_racing/logic.lua` (game-specific career update) |
| `create_race` | Entity | `games/horse_racing/logic.lua` (game-specific) |
| `can_unlock_slot` | Action | `games/horse_racing/logic.lua` (game-specific) |

**Logic for "game-specific" vs "engine":**
- `simulate_race` is game-specific because it encodes horse physics (energy drain,
  speed curves, temperament variance). BattleBots will have a different `resolve_fight`.
  The shared pattern is `scores_to_odds` in `engine/primitives/resolution.lua`.
- `settle_bets` is engine-level because the Win/Place/Show pattern is genre-independent.
  Any betting game uses the same settlement logic.

## Consequences

**Positive:**
- Every function in the studio has a permanent home determined by its primitive type
- Snake, BattleBots, and future games call engine functions instead of re-implementing
- Claude can determine a function's primitive type from its name alone
- The Phase 4 linter has a mapping table to validate against

**Negative:**
- Existing `horse_racing/logic.lua` needs to be refactored to use engine imports
- Migration of shared functions (genetics, odds, market) requires updating all callers

**Migration is NOT in scope for Phase 2e or Phase 3.** This ADR locks the target
state. The migration directive (Phase 2f) executes it before Snake is built.
Snake is the proof that the migration worked.
